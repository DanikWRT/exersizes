# Workout_program — Архитектура

Назначение: описать, как устроены данные (сущности, связи, JSON-файлы) и ключевые принципы MVP.

## Хранилище и файлы
Используем локальные JSON-файлы (без БД):
- `app/data/exercises.json` — справочник упражнений
- `app/data/logs.json` — журнал подходов
- `app/data/plans.json` — планы недель

Принцип: это “файловая база данных” для MVP — просто, прозрачно, переносимо.

## Сущности данных

### Exercise (упражнение)
Файл: `app/data/exercises.json`

Назначение: единый справочник, чтобы логировать подходы по одному и тому же названию.

Поля (MVP):
- `id` — строковый идентификатор (slug), производный от имени
- `name` — каноническое имя (то, как хранится в логах)
- `muscle_group` — группа мышц (строка)
- `aliases` — список альтернативных названий

Поиск на вводе:
- нормализуем ввод (`strip()` + `casefold()`)
- ищем совпадение по `name` и по `aliases`

### LogEntry (запись о подходе)
Файл: `app/data/logs.json`

В проекте это основная единица учёта: **один подход = одна запись**.

Поля (MVP):
- `date` — `YYYY-MM-DD`
- `exercise` — ссылка на упражнение
  - в MVP храним каноническое имя (`Exercise.name`), чтобы файл был читаем человеком
- `weight` — число (кг)
- `reps` — целое
- `notes` — опциональная строка
- `source` — опционально (например `raw_import.md`)

### WeekPlan (план недели)
Файл: `app/data/plans.json`

Назначение: зафиксировать “фокус дня” на 7 дней от даты старта.

Поля (MVP):
- `start` — дата начала недели `YYYY-MM-DD`
- `days` — массив из 7 элементов:
  - `date`
  - `focus` — строка (например `legs+shoulders`)

## Связи между сущностями
- `LogEntry.exercise` соответствует `Exercise.name`.
  - На вводе допускаем алиасы, но сохраняем каноническое имя.
- `WeekPlan` не привязан к упражнениям напрямую: он задаёт фокус/сплит дня.

## Принципы

### Почему JSON
- быстрый MVP без инфраструктуры
- легко бэкапить/синкать (копия папки `app/data/`)
- можно читать/править вручную при необходимости

### Нормализация и согласованность данных
- имена упражнений: `strip()` + `casefold()`
- числа: `weight` → float, `reps` → int
- борьба с дубликатами: алиасы + проверка по нормализованному имени

### Конфликты и риски записи
- Параллельные запуски CLI могут приводить к конфликтам записи (race condition).
- В MVP это не решается полностью; рекомендуемый режим — один пользователь/один процесс.
- При расширении: можно добавить файловые блокировки или переход на SQLite.

### Как расширять (после MVP)
- Добавлять новые поля **с сохранением обратной совместимости** (старые записи без поля должны быть валидны).
- При усложнении схемы — ввести версионирование данных и/или миграции.
- Если потребуется “тренировка как сущность” — добавить сущность Session/Workout и связывать LogEntry с ней.
